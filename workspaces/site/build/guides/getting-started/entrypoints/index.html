Adding a new entrypoint

What if we want to add another script to our project, which should be built and output separately? Let's try it.

Add a second scriptâ€‹

Add a new script to your project at src/landing.js.

src/landing.js
console.log('landing page script')

To build it, specify the new script in bud.config.js using bud.entry.

  • Javascript
  • TypeScript
bud.config.js
module.exports = bud =>
  bud.entry('landing', 'landing.js').template()
bud.config.js
import type {Framework} from '@roots/bud'

export default function (bud: Framework): void {
  bud.entry('landing', 'landing.js').template()
}

Two steps forward, one step backâ€‹

If you run a build now you'll see your new file was compiled, but you might also notice that dist/main.js is missing.

Until now, we'd been successfully relying on bud's implicit inclusion of src/index.js, which it will attempt to use if bud.entry is not called.

But now that we are specifying an entrypoint in our config, we'll need to explicitly specify src/index.js as well.

  • Javascript
  • TypeScript
bud.config.js
module.exports = bud =>
  bud
    .entry('main', 'index.js')
    .entry('landing', 'landing.js')
    .template()
bud.config.js
import type {Framework} from '@roots/bud'

export default function (bud: Framework): void {
  bud
    .entry('main', 'index.js')
    .entry('landing', 'landing.js')
    .template()
}

Adding files to an existing entrypointâ€‹

Currently, we are importing our application css by importing it into the src/index.js entrypoint.

src/index.js
import './index.css'

console.log('hello, world.')

This is a totally valid way to handle css in your build, but let's instead specify the css file in bud.config.js so we can keep learning about how to work with it.

Start by removing the css import from src/index.js

src/index.js
console.log('hello world!')

Now, add the css to the main entry.

  • Javascript
  • TypeScript
bud.config.js
module.exports = bud =>
  bud
    .entry('main', ['index.js', 'index.css'])
    .entry('landing', 'landing.js')
    .template()
bud.config.ts
import type {Framework} from '@roots/bud'

export default function (bud: Framework): void {
  bud
    .entry('main', ['index.js', 'index.css'])
    .entry('landing', 'landing.js')
    .template()
}

As you can see, bud.entry's second parameter can be either a string or an array, and we're using the array so we can specify multiple files.

Specifying entrypoints as an objectâ€‹

While we can call bud.entry as many times as we need, we can also specify all of our entrypoints in one call using object syntax.

bud.config.js
module.exports = bud =>
  bud
    .entry({
      main: ['index.js', 'index.css'],
      landing: 'landing.js',
    })
    .template()

Subsequent calls to bud.entry will merge our existing entrypoints with our new ones, rather than overwriting everything. So this is safe and functionally equivalent:

bud.config.js
module.exports = bud =>
  bud
    .entry({main: ['index.js', 'index.css']})
    .entry({landing: 'landing.js'})
    .template()
